0. 摘要

命中率只是中间变量。真正该看的指标是：满足 SLO 的 goodput、P50/P95 TTFT、TPOT 稳定性、GPU 空转、单位有效 token 成本，以及跨请求复用的安全性。

KV 复用远不止"缓存命中"这么简单。它本质上把本来在 prefill 阶段烧掉的计算，转化成了一个状态传输与控制问题：这段状态兼不兼容、现在在哪儿、要多久才能变成推理引擎能直接绑定的 KV page/block，以及负载一上来是不是干脆放弃复用、直接重算更划算。

所以这份报告的焦点放在 传输层：外部 KV store、多层级物化、直接内存通路、CXL 类内存池、prefill/decode 解耦后的 KV 交接，以及决定这些状态操作是否划算的路由与元数据控制面。

3. 为什么 KV reuse 变成了系统问题

KV-cache 的工作重心已经从本地内存布局扩展到面向传输的系统问题：可复用状态开始跨越服务、层级、主机、机架，甚至模型和 workflow 的边界。下图是范围内语料的信号图，不是 bibliometric claim。
图 1. 2024 年之后，KV 不再只是本地显存管理；它进入了 store、direct path、routing policy、CXL-like memory 和 agentic workflow 的视野。

本地底座

Paged KV、radix cache、prefix cache。KV 对象仍主要留在 engine 内部。

状态服务

外部 KV pool、分布式 metadata、hot/warm/cold tier，让复用成为一种服务。

传输底座

RDMA、GPUDirect、NIXL、NVMe-oF、CXL-like semantics 决定物化的真实成本。

7. 请求路径：每一次命中都意味着一次物化义务

传输不是调度之后的附属章节。一个迟到、碎片化、CPU-bound 或不安全的 hit，可能名义上提高了命中率，实际上却拉低了 goodput。
1. 暴露应用区分 stable prompt、tool schema、workspace memory、dynamic tail、tenant policy 和 reuse hint。
2. 查找Router 对 token block/chunk 做 hash；metadata plane 检查 model、tokenizer、layout、RoPE、dtype、TTL 和 owner。
3. 决策在 SLO 和队列状态下选择 route、fetch、prefill、repair、demote 或 recompute。
4. 物化准备 memory descriptor、批处理碎片、选择 tier/path、异步传输并报告 completion。
5. Decodelive decode 消费 resident KV 并追加新 KV；远端逐 token fetch 通常不可接受。
6. 发布admit、replicate、compress、evict、demote 或发布新 metadata 供后续请求使用。
图 4. 请求路径：一次 cache hit 只有经过 lookup、compatibility checking、action selection、materialization、decode binding 和 lifecycle publication 之后，才真正变成可用的收益。
Decode 的微妙之处。 KV 复用主要节省 prefill。decode 的收益更多是间接的：减少 prefill 与 decode 的资源竞争、把请求放在所需状态附近、降低 live working set 压力，以及避免把大上下文传输塞进逐 token 路径。
SLO 税。 这里指为了物化可复用状态而引入的 lookup、queueing、descriptor setup、transfer、validation、retry 和 tail-latency risk。

8.2 命中率侧：可达 KV 池如何改变 prefill 分母

图 5. 命中率直觉：扩大有效可达 KV 池，可以把 agentic trace 中原本在本地 miss 掉的重复 prefix 转化为可复用的 prefill 节省。

解读

在 Codex-style agentic trace 上，仅靠本地或 system-prompt cache 可能错过大部分可复用状态。分布式池扩大了跨 turn、跨 instance 的可达 working set，使得系统经常只需 prefill 新增的 delta，而不必重新计算整个累积上下文。

一个 prefill-only 的直觉：如果某一 turn 的 90% 是可复用 prefix，那么 h=1.7% 时仍需做约 98.5% 的 prefill 工作；h=92.2% 时只剩约 17.0%。这给出约 5.8x 的理论 prefill-work 缩减，实际报告的吞吐提升为 3.8x，说明命中率确实不是终点。

8.3 搬运侧：tier 越深，搬运税越容易吃掉收益

解读

搬运图使用同一个 3.8 GB / 100K-token Kimi-style 对象。柱高来自峰值带宽除法，是下界，不是测得 TTFT。真实系统还要付 block lookup、transfer setup、registration、scatter-gather 碎片化、endpoint selection、retry、interference 和 tail queueing 的代价。

因此 tier depth 必须和 policy 绑定。更多存储可以提高 p_hit，但也可能推高 p_fetch 和 transfer_cost；只有 route-to-state、prefetch、batching 和 direct path 把对象移出请求 critical path 时，命中才真正有收益。

图 6. 搬运下界：更大、更深的 KV 池提高了复用机会，但传输路径的延迟和方差足以抹掉命中收益。

10. Mooncake TE：transfer-plan substrate，不只是 RDMA wrapper

Mooncake TE 的关键不只是"能走 RDMA"。RDMA 只是一个 backend；更高层的贡献是 registered segments、异步 BatchTransfer、拓扑感知 path selection，以及可被 Mooncake Store 或 vLLM connector 组合成 KV 物化方案的底座能力。

它在 raw network transport 之上增加了什么

  • Segments：覆盖 RAM、VRAM 和 NVMe-oF-like storage endpoints 的命名地址空间。
  • BatchTransfer：对非连续区域发起异步 read/write batches，避免每个碎片走独立的慢路径。
  • Backend diversity：文档覆盖 TCP、RDMA/GPUDirect RDMA、EFA、NVMe-oF、NVLink、HIP 等路径。
  • Path management：NIC preference、多 NIC pooling、大传输 slicing、endpoint pooling、retry 和 completion semantics。

vLLM integration 使用 Mooncake master 管理 KV-block metadata 和 client health，由 workers 执行后台搬运，并在可用时用 GPUDirect RDMA 实现不占 SM 的 zero-copy KV transfer。S1S2

11. Beluga：CXL 是底座，Beluga-KVCache 才是 KV-aware 层

Beluga 最清晰的读法是两步：先指出 RDMA-style movement 对碎片化 KV 不友好；再在通用 CXL memory semantics 之上加入 KV-aware indexing、copy 和 control。CXL pooling 本身并不理解 KV block。
图 8. Beluga 的 framing：CXL 提供 shared-memory semantics；KV awareness 来自其上的 index、layout handling、copy kernels 和 scheduling。

Beluga 对 RDMA 的批评

Beluga 认为 RDMA 是网络协议而非内存总线。CPU-driven path 可能经过 bounce buffer；GPU-driven path 可能需要 polling kernel 和同步；碎片化 KV layout 又会压迫 NIC scatter-gather 上限。论文给出了 Qwen-32B GQA 的具体例子：每 token 有 128 个非连续 chunks，更稀疏的情况可能产生更多 fragments。S8

Beluga 对 CXL pooling 的补足

CXL switch-attached memory 提供 shared-memory semantic，但它不知道一个 KV 对象的 model、tokenizer、RoPE position、block layout 或 tenant policy。Beluga-KVCache 增加了 KVCache index、data copy kernels、shared-memory RPC path 和 scheduler simplification，才让 CXL pool 变成 KV-aware。

问题Beluga 的回答实际约束
为什么需要 memory semantics?load/store-like access 可以去掉一部分 verbs/queue/completion overhead；论文报告了相对 RDMA baseline 的 microbenchmark latency gains。CXL bandwidth、switch contention、coherence scope、adapter count、GPU-to-CXL path 成熟度仍是部署约束。
为什么不能只说 CXL pooling?KV 仍然需要 block metadata、layout conversion、compatibility checks 和 lifecycle policy。KV awareness 是上层软件，不是 memory fabric 的原生功能。
为什么不能替代所有 RDMA?Beluga 最适合作为 nearby shared memory，服务低延迟 KV access。更大范围的 cluster 和 cross-cluster movement 仍然更自然地落在 RDMA/网络传输上。

Case-study 视角

ProblemKV 大、碎片化、跨 tier 访问频繁时，RDMA-style movement 会变得笨重。
Boundary movedremote memory pool 变成 CXL memory-semantic pool，并叠加 KV-aware software layer。
Data pathCXL-attached memory semantics 加 KV index、gather/scatter-oriented access、copy kernels 和 RPC/control paths。
Control dependencyraw CXL memory 之上的 placement、memory allocation、layout conversion、block indexing 和 scheduler integration。
Evidencepreprint-level system proposal and evaluation；作为架构方向有价值，但仍依赖 CXL 生态成熟度。
Best fitnearby pod/rack memory expansion，且 low-latency shared memory semantics 能胜过显式搬运开销。

Not a claimCXL pooling 不会自动 KV-aware；Beluga 是在底座上方加入 KV semantics。

12. Dynamo / NIXL：聚焦 direct handoff，而非完整 KV 生命周期

Dynamo 的传输强项是生产化 disaggregated serving 中基于 NIXL 的 direct VRAM-to-VRAM KV transfer。它最适合在快速 NVIDIA-oriented fabric 内完成 prefill 到 decode 的 KV 交接。
图 9. Dynamo/NIXL 的核心价值是生产化 disaggregated serving 路径中 direct, coordinated prefill-to-decode KV handoff。

Dynamo 文档把 disaggregated execution 描述为：prefill 计算 KV、KV transfer 到 decode、decode 生成 token。文档说明 NIXL 可以把 KV 直接从 prefill VRAM 传到 decode VRAM，并且 transfer 是 nonblocking 的。Router 使用 cache overlap 和 load 信号，注入 backend-specific transfer metadata，并在 fabric backends 上协调传输。S7

维度Mooncake TE / StoreDynamo / NIXL
定位KV-centric distributed store 加 multi-substrate transfer substrate。production disaggregated serving stack 中的 direct KV handoff。
最佳路径HBM/DRAM/NVMe-oF-like segments、RDMA/TCP/GDS-like movement、distributed pool。prefill VRAM 到 decode VRAM，在快速 GPU/network fabric 内 nonblocking transfer。
强项更宽的 external KV lifecycle：cache pool、offload、cross-instance prefix reuse。与 NVIDIA GPU/fabric ecosystem 深度对齐，handoff 编排紧密。
潜在缺口transport 之上需要更多 policy 和 governance。除非叠加其他 cache/tiering 层，不等于 general-purpose long-lived multi-tier KV store。

Case-study 视角

Problem在 PD serving 中，prefill 产生的 KV 必须快速到达 decode，不能阻塞 critical path，也不能消耗不必要的 GPU compute。

13. PrfaaS：跨集群 prefill 是有条件的边界外移

PrfaaS 的意义在于追问 PD boundary 能外移多远。答案不是"越远越好"：remote prefill 只有在模型侧 KV 缩减、selective admission、bandwidth-aware routing 和 tail-risk control 同时成立时才划算。
图 10. PrfaaS 图强调跨集群 prefill 的条件性：返回状态必须足够小，且远端端到端路径必须在 tail-risk 约束下胜过本地 prefill。

PrfaaS 的出发点很明确：传统 dense-attention PD 往往被绑定在高带宽域内，因为 prefill 产生的 KV 太大。Hybrid 或 KV-efficient attention 可以缩小返回状态，使 commodity Ethernet 或跨集群服务对某些 long-context 请求变得可行。S9

条件为什么重要PrfaaS 机制
长 uncached prefill短请求省下的计算不足以支付跨集群传输成本。基于长度的 threshold routing。
返回 KV 足够小dense model 可能输出太多 KV，不适合 commodity Ethernet。KV-efficient hybrid attention 和模型侧缩减。
Bandwidth-aware admission跨集群链路会波动并成为瓶颈。拥塞积累前做 bandwidth-aware scheduling。
Cache-aware placementprefix cache 分布不均会抹掉 remote prefill 收益。global KV manager 和 hybrid prefix cache。
remote_prefill 划算的条件： saved_local_prefill_time > remote_queueing + remote_prefill_time + KV_transfer_time + metadata_control_overhead + tail_risk + fallback_risk
这是一条 operator decision rule。PrfaaS 不是说 remote prefill 普遍更好，而是说当返回状态足够小、网络路径受控时，它可以成为一种选择性动作。

Case-study 视角

Problemprefill 需求 bursty 且异构；本地 cluster 可能被 decode pressure 占住，而远端 capacity 可吸收长 prefill。
Boundary movedPD 边界从 intra-cluster 扩展到 selected requests 的跨集群/跨站点 prefill service。
Data path远端 prefill 加回传 KV/state，经由 less specialized links；前提是模型侧 KV 缩减和带宽可行。
Control dependencylength threshold、bandwidth-aware admission、global KV manager、cache placement 和 tail-risk fallback。
Evidencepreprint-level 部署架构提案；在 selective long-context admission 下更有说服力。
Best fit长 uncached prompts、bursty prefill、异构集群、KV-efficient 模型架构。
Primary stress pointWAN/跨集群排队、带宽波动、回传状态大小，以及 remote path miss SLO 时的恢复。
Not a claimremote prefill 并不总是更好；只有不等式成立时才应选择。

15. 评估纪律与治理考量

每一条 transport claim 都应回答一个 operator 的问题：考虑搬运、控制、质量和治理成本之后，这个机制是否提高了满足 SLO 的 goodput？

评估维度

挪动的边界local page、external store、PD handoff、CXL pool、跨集群 service 或 agent workflow。
Workload regimeagentic turn traces、RAG/document reuse、long-output decode、multi-tenant prefixes 或 synthetic stress。
命中分母request-level hit、block-level hit、byte-weighted hit 或 prefill-compute-weighted hit。
路径指标mean/p95 transfer time、queueing、CPU overhead、GPU interference、descriptor setup、retry rate。
终局指标P50/P95 TTFT、TPOT stability、SLO goodput、cost per token、GPU utilization、quality retention。

治理 checklist

Tenant isolationmetadata 必须编码 tenant、ACL、policy 和 revocation behavior。
Compatibility safetymodel/tokenizer/RoPE/dtype/layout 不匹配时拒绝复用，除非 repair 已被显式验证。
Provenance记录 producer、source context、validity epoch、quality confidence 和 fallback path。
StalenessTTL、version 和 context mutation 应使 reuse confidence 失效或降级。
Auditability每个 reused KV object 都留下 lookup、decision、transfer、bind、repair、fallback trace。
Threat model 示例。 某个 KV object 可能技术上可达，却因为来自不同 tenant 或 model version 而语义上不安全。传输层搬得越快，越需要 compatibility metadata、tenant policy、provenance 和 recomputation fallback 作为硬门。
