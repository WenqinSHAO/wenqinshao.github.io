0. Abstract

Hit rate is an intermediate variable. Serving operators ultimately care about SLO-qualified goodput, P50/P95 TTFT, TPOT stability, GPU idle time, cost per useful token, and safe cross-request reuse.

KV reuse converts prefill compute into a transport and control problem. A reusable prefix or block is valuable only after the serving stack answers four questions: is the state compatible, where is it, how fast can it become engine-ready, and what is the fallback if the answer changes under load?

This report focuses on the transport-facing layer of KV reuse: external KV stores, multi-tier materialization, direct memory paths, CXL-like pooling, disaggregated prefill/decode handoff, and routing/control metadata that determine whether saved prefill remains visible as lower TTFT or higher goodput.

1. Data scope and methodology

This is a transport-focused technical report, not a comprehensive academic survey. It selects systems and papers that change how KV state is placed, moved, materialized, routed, or governed across service and memory boundaries.

Scope boundary

In scopeExternal KV stores, prefill/decode handoff, distributed KV transfer, CXL or remote-memory pooling, direct GPU paths, KV-aware routing, metadata/ABI requirements, and transport-visible governance.

Evidence handling

3. Why this became a systems topic

KV-cache work has moved from local memory layout into a transport-facing systems problem: reusable state now crosses service, tier, host, rack, and sometimes model/workflow boundaries. The figure is a scoped signal map, not a bibliometric claim.
Figure 1. A full-width signal map makes the branch-out visible: after 2024, KV work is no longer only allocation and prefix caching; it reaches stores, direct paths, routing policy, CXL-like memory, and agentic workflows.

Local substrate

Paged KV, radix caches, prefix cache. The object is still engine-local.

State service

External KV pool, distributed metadata, and hot/warm/cold tiers make reuse a service.

Transport substrate

RDMA, GPUDirect, NIXL, NVMe-oF, and CXL-like semantics decide materialization cost.

Control plane

Cache-aware routing, load balancing, SLO admission, and future-use hints decide the action.

Workflow boundary

Agent turns, tool stalls, shared workspaces, and cross-model compatibility widen reuse.

Earlier KV work mainly optimized live local memory: allocation, paging, retention, and repeated prompt fragments. Recent work makes KV cross service boundaries: prefill to decode, engine to external store, GPU to DRAM or SSD, pod to pod, and in some proposals model or agent to model or agent.

6. Structural lens: compile reusable context into SLO-aware state actions.

The cross-layer story is not a loose ecosystem map. It is a compilation path: application and model structure produce candidate KV objects; the controller validates and chooses actions; memory and transport substrates make those actions feasible or too expensive.
Figure 3. Structural lens: application-visible reusable context and model state shape are compiled into concrete state actions by a KV-aware controller; substrates constrain which action is actually cheap.

7. Request journey: every hit creates a materialization obligation.

Transport is not a postscript after scheduling. It constrains which cache-aware decision is good because a late, fragmented, CPU-bound, or unsafe hit can reduce goodput while increasing nominal hit rate.
1. ExposeApplication separates stable prompt, tool schema, workspace memory, dynamic tail, tenant policy, and reuse hints.
2. LookupRouter hashes token blocks or chunks; metadata plane checks model, tokenizer, layout, RoPE, dtype, TTL, and owner.
3. DecideChoose route, fetch, prefill, repair, demote, or recompute under SLO and queue state.
4. MaterializePrepare memory descriptors, batch fragments, select tier/path, run async transfer, and report completion.
5. DecodeLive decode consumes resident KV and appends new KV; remote per-token fetch is usually unacceptable.
6. PublishAdmit, replicate, compress, evict, demote, or publish new metadata for future turns.
Figure 4. Request journey: a cache hit becomes useful only after lookup, compatibility checking, action selection, materialization, decode binding, and lifecycle publication.
Decode nuance. KV reuse mainly saves prefill. Decode benefits indirectly when reuse keeps prefill from competing with decode, places the request near needed state, reduces live working-set pressure, and avoids transferring large context into the token-by-token path. For long-output workloads, decode is a retention and bandwidth problem more than a classic prefix-hit problem.
SLO tax. In this report, SLO tax means the extra lookup, queueing, descriptor setup, transfer, validation, retry, and tail-latency risk introduced by materializing reusable state.

8.2 Hit-rate side of the trade-off

Figure 5. Hit-rate intuition: increasing the effective reachable KV pool can convert repeated agentic prefixes from mostly missed local state into reusable prefill savings.

Interpretation

On Codex-style agentic traces, a local/system-prompt-only cache can miss most of the reusable state. A distributed pool increases the reachable working set across turns and instances. This changes the denominator of prefill work: the system now often prefill-computes only the new delta instead of the entire accumulated context.

A simple prefill-only intuition: if 90% of a turn is reusable prefix, then hit rate h=1.7% leaves about 98.5% of prefill work, while h=92.2% leaves about 17.0%. That is roughly 5.8× theoretical prefill-work reduction before queueing, decode, and transfer overheads. The reported 3.8× throughput gain is therefore plausible and also shows that hit rate is not the final metric.

8.3 Movement side of the trade-off

Interpretation

The movement plot uses the same 3.8 GB 100K-token Kimi-style object. Bars are lower bounds from peak bandwidth division, not measured TTFT. Real systems add block lookup, transfer setup, registration, scatter-gather fragmentation, endpoint selection, retries, interference, and tail queuing.

This is why tier depth has to be coupled with policy. More storage can raise p_hit, but it also raises p_fetch and transfer_cost unless route-to-state, prefetch, batching, and direct paths keep the object out of the request's critical path.

The transport layer cannot optimize anonymous tensor bytes. It needs a minimal KV object contract that exposes identity, compatibility, location, validity, policy, and completion semantics.

KV needs names at block/chunk/session scope before it can be routed, fetched, or shared.

Transfer plan

Layer/head/page fragments become batched scatter-gather plans with async completion and fallback semantics.

10. Mooncake TE as a transfer-plan substrate.

Mooncake TE is best read as more than RDMA. RDMA is one backend; the higher-level contribution is registered segments, asynchronous batch transfers, topology-aware path selection, and a substrate that Mooncake Store or vLLM connectors can compose into KV materialization.

What it adds above raw network transport

  • Segments: named address spaces over RAM, VRAM, and NVMe-oF-like storage endpoints.
  • BatchTransfer: asynchronous read/write batches over non-contiguous regions.
  • Backend diversity: documented TCP, RDMA/GPUDirect RDMA, EFA, NVMe-oF, NVLink, and HIP paths.
  • Path management: NIC preference, multi-NIC pooling, slicing of larger transfers, endpoint pooling, retry and completion semantics.

The vLLM integration then uses the Mooncake master for KV-block metadata and client health, workers for background movement, and GPUDirect RDMA for SM-free, zero-copy KV transfer where applicable. S1S2

11. Beluga: CXL is the substrate; Beluga-KVCache is the KV-aware layer.

Beluga is clearest when framed in two steps: it argues that RDMA-style movement is awkward for fragmented KV, and it adds KV-aware indexing/copy/control on top of generic CXL memory semantics. CXL pooling alone is not KV-block aware.
Figure 8. Beluga framing: CXL provides shared-memory semantics, while KV awareness is supplied by indexing, layout handling, copy kernels, and scheduling above the substrate.

What Beluga adds on RDMA

Beluga argues that RDMA is a network protocol rather than a memory bus. CPU-driven paths can copy through bounce buffers, GPU-driven paths can require polling kernels and synchronization, and fragmented KV layouts can stress NIC scatter-gather limits. The paper gives a concrete Qwen-32B GQA example with 128 non-contiguous chunks per token, and sparser cases can create even more fragments. S8

What Beluga adds on CXL pooling

CXL switch-attached memory gives a shared memory semantic, but it does not know the model, tokenizer, RoPE position, block layout, or tenant policy of a KV object. Beluga-KVCache adds the KVCache index, data copy kernels, shared-memory RPC path, and scheduling simplification needed to make the CXL pool KV-aware.

12. Dynamo / NIXL: focused direct handoff rather than a full KV lifecycle.

Dynamo's transport strength is production disaggregated serving with NIXL-based direct VRAM-to-VRAM KV transfer. It is strongest when KV is handed from prefill to decode inside a fast NVIDIA-oriented fabric.
Figure 9. Dynamo/NIXL framing: the core value is direct, coordinated prefill-to-decode KV handoff inside a production disaggregated serving path.

The Dynamo documentation describes disaggregated execution as prefill computing KV, KV transfer to decode, and decode token generation. It states that NIXL transfers KV directly from prefill VRAM to decode VRAM and that the transfer is nonblocking. The router uses cache overlap and load, injects backend-specific transfer metadata, and coordinates transfer over fabric backends. S7

13. PrfaaS: cross-cluster prefill is a conditional boundary shift.

PrfaaS matters because it asks how far the PD boundary can move. The answer is not always outward: remote prefill wins only when model-side KV shrinkage, selective admission, bandwidth-aware routing, and tail-risk control line up.
Figure 10. A larger PrfaaS diagram emphasizes the conditional nature of cross-cluster prefill: the returned state must be small enough and the end-to-end remote path must beat local prefill under tail-risk constraints.

PrfaaS starts from a clear condition: conventional dense-attention PD is tied to a high-bandwidth domain because prefill emits too much KV. Hybrid or KV-efficient attention can shrink the returned state and make commodity-Ethernet or cross-cluster service plausible for selected long-context requests. S9

15. Evaluation discipline and governance considerations

Every transport claim answers an operator question: does this mechanism improve SLO-qualified goodput after movement, control, quality, and governance costs?

Evaluation dimensions

Boundary movedLocal page, external store, PD handoff, CXL pool, cross-cluster service, or agent workflow.
Workload regimeAgentic turn traces, RAG/document reuse, long-output decode, multi-tenant prefixes, or synthetic stress.
Hit denominatorRequest-level hit, block-level hit, byte-weighted hit, or prefill-compute-weighted hit.
Path metricMean/p95 transfer time, queueing, CPU overhead, GPU interference, descriptor setup, retry rate.
End metricP50/P95 TTFT, TPOT stability, SLO goodput, cost per token, GPU utilization, quality retention.

Governance checklist

Tenant isolationObject metadata must encode tenant, ACL, policy, and revocation behavior.
Compatibility safetyReject on model/tokenizer/RoPE/dtype/layout mismatch unless repair is explicitly validated.
ProvenanceStore producer, source context, validity epoch, quality confidence, and fallback path.
StalenessTTL, version, and context mutation invalidate or downgrade reuse confidence.
AuditabilityEach reused KV object leaves a trace: lookup, decision, transfer, bind, repair, fallback.
